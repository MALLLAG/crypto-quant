# 스펙 주도 개발 (Spec Driven Development)

기능 구현 전에 테크스펙과 구현 계획 문서를 체계적으로 작성하도록 도와줍니다.

---

## 핵심 원칙

- **테크스펙 먼저**: 코드는 테크스펙의 구현일 뿐이다. 테크스펙 없이 코드를 작성하지 마라.
- **최소 변경**: 변경은 필요한 부분만. 모든 변경을 최대한 단순하게. 최소한의 코드 영향.

---

## 워크플로우

1. **테크스펙 작성**: 기능의 요구사항, 설계, 기술적 결정사항을 문서화
2. **구현 계획 작성**: 테크스펙 기반으로 단계별 구현 계획과 체크리스트 생성

## 파일 구조

```
docs/spec/
├── 001-기능명/
│   ├── TECH-SPEC.md           # 테크스펙
│   └── PLAN.md                # 구현 계획
└── ...
```

## 진행 방법

### 테크스펙 작성

1. 사용자로부터 기능 요구사항 청취
2. `docs/spec/` 디렉토리의 기존 폴더 번호 확인하여 다음 번호 결정
3. [TECH-SPEC.template.md](./TECH-SPEC.template.md) 기반으로 `TECH-SPEC.md` 작성
4. 사용자와 함께 요구사항 및 설계 검토

### 구현 계획 작성

> **중요**: 테크스펙 완성 후 반드시 사용자에게 PLAN 작성 허락을 받아야 합니다.

1. 테크스펙 완성 후 사용자에게 PLAN 작성 여부 질문
2. **사용자 승인 후에만** [PLAN.template.md](./PLAN.template.md) 기반으로 `PLAN.md` 작성
3. 각 단계별 체크리스트와 의존성 명시

### 구현 진행

1. 구현 계획 문서의 체크리스트 확인
2. 각 작업 완료 시 체크 표시 (`- [x]`)
3. 단계 완료 시 상태 테이블 업데이트

---

## 디렉토리 규칙

- 번호: 3자리 (001, 002, ...)
- 기능명: kebab-case
- 예시: `docs/spec/001-user-authentication/TECH-SPEC.md`

---

## 주의사항

1. **테크스펙 먼저**: 구현 계획은 반드시 테크스펙 완성 후 작성
2. **사용자 승인 필수**: PLAN 문서는 사용자가 명시적으로 허락할 때만 작성
3. **점진적 상세화**: 처음부터 완벽할 필요 없음, 점진적으로 보완
4. **프로젝트 원칙 준수**: CLAUDE.md의 함수형 DDD 원칙 따르기
